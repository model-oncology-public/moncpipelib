# moncpipelib: What's New in v0.32.0

*Headline: psycopg2 -> psycopg3 driver migration (Migration 014).*

## Breaking Changes to Call Out

1. **`psycopg2-binary` is no longer a dependency.** Consumers that imported `psycopg2.extensions.connection` directly (e.g. for type annotations) need to update to `psycopg.Connection`. The public moncpipelib API (`PostgresResource`, `PostgresIOManager`, `PostgresPolarsSchema`, writer helpers) is **unchanged** — `database.write(df, target=..., context=...)` looks identical from the consumer side.
2. **SQLAlchemy dialect URL changed** from `postgresql+psycopg2` to `postgresql+psycopg`. Consumers building their own SA engines should use `engine.dialect.driver == "psycopg"` to assert the cutover.
3. **SQLAlchemy reads need explicit Loader registration.** Consumer code that uses `engine.connect()` directly for reads (rather than going through `PostgresResource.read_*`) must call `PostgresPolarsSchema.register_uuid_adapter_sa(sa_conn)` and `register_json_adapters_sa(sa_conn)` after `engine.connect()` if it expects JSONB / UUID columns to come back as `pl.String`. Without registration, the SA-managed psycopg3 connection's default loaders return parsed Python objects (`dict` for JSONB, `uuid.UUID` for UUID), which breaks Polars schema inference. The resource's own read helpers (`read_streaming`, etc.) already handle this internally.

## Driver Migration (Migration 014)

The full migration spans seven phases (A-G) and is documented at [`docs/migrations/20260507_266-psycopg3-migration.md`](../migrations/20260507_266-psycopg3-migration.md). Highlights for consumers:

- **Modern adapter API.** UUID / JSON / JSONB string-passthrough now uses `psycopg.adapt.Loader` subclasses instead of `psycopg2.extras.register_default_json` + `psycopg2.extensions.new_type`. The contract — JSONB and UUID columns come back as `str`, not parsed Python objects — is preserved.
- **Streaming COPY.** `cursor.copy_expert(sql, BytesIO)` was replaced with `with cursor.copy(sql) as copy:` plus per-chunk `copy.write(...)`. **One COPY statement** spans an entire chunked write, vs psycopg2's pattern of N separate `copy_expert` calls (one per chunk). This is a transactional-semantics improvement; the new shape's atomicity boundary matches the natural unit of work.
- **`cursor.executemany` replaces `psycopg2.extras.execute_values`.** Same intent (bulk parameterized INSERT), now via psycopg3's pipelined extended-query protocol. The `BulkInsertMethod.EXECUTE_VALUES` enum value is preserved for back-compat — the string names the *intent*, not the underlying driver function.
- **No public API change.** `PostgresResource.write(...)`, `PostgresIOManager.handle_output(...)`, `BatchedDataFrame`, `make_contract_checks`, and the contract YAML schema are all unchanged.

## Performance

We measured the new driver against the old one on real Azure flex-server hardware (`pg-nonprod / analytics`, nonprod). Full methodology and per-iteration data are in [`docs/migrations/20260507_266-psycopg3-migration.md` -> "Empirical performance results"](../migrations/20260507_266-psycopg3-migration.md#empirical-performance-results). Headline numbers:

- **Raw COPY 1M rows × 10 cols** (production-shape, fresh connection per write):
    - Median wall clock: **5.85s -> 3.25s (-44%)**.
    - Tail rate (iters >= 10s): **6/19 -> 3/19 (-50%)**.
    - Peak Python heap: 5.87 MiB -> 11.12 MiB (a ~5 MiB regression from psycopg3's internal copy-protocol buffer; bounded; harmless absolute).
- **SCD2 100k rows, 50% mutation** (production-shape, fresh connection per write):
    - Both drivers post **0/19 iters >= 10s** — the multi-statement-per-write pattern (staging COPY + count + UPDATE expire + diff CTAS + final INSERT + commit) amortizes per-connection backend warmup so completely that the long tail disappears entirely.
    - Median is statistically a wash (psycopg3 ~21% slower, well within iter-to-iter variance) but psycopg3 has **half the variance** (stdev 0.57s vs 1.02s) and a tighter max (4.58s vs 7.50s).

Net: psycopg3 is materially better on raw bulk COPY in production shape and equivalent-with-tighter-variance on the SCD2 path that real pipelines exercise.

## Memory Footprint

Peak Python heap during writes is bounded by chunk size in both drivers (psycopg2 already chunked since #245). psycopg3's `cursor.copy()` adds ~5 MiB of internal buffer overhead vs psycopg2's `copy_expert(BytesIO)` direct read on the same chunk — a regression in relative terms but small in absolute terms (11 MiB vs 6 MiB on a 95 MiB DataFrame). For the executemany path psycopg3 is actually ~13% lower on peak heap.

## Known Operational Footgun: SCD2 Indexes

While verifying the SCD2 path, an SCD2 target table without a partial index on `(business_key) WHERE is_current = true` produced a **416-second merge for 100k rows** — a sequential-scan O(N*M) pathology. With the index, the same merge ran in ~3.5s (a ~120x speedup). This is **not a moncpipelib regression** — production SCD2 tables already carry these indexes — but is worth calling out: if you create a new SCD2 target, mirror the production index profile:

```sql
CREATE INDEX idx_<table>_bk_current
  ON <schema>.<table> (business_key) WHERE is_current = true;
CREATE INDEX idx_<table>_bk_eff
  ON <schema>.<table> (business_key, effective_from);
```

## Compliance

`SECURITY.md` was updated in Phase G to document psycopg3 as the supported PostgreSQL data-plane driver. TLS handling, parameterized queries, JSON-as-string passthrough, and audit-trail behavior are all preserved across the swap and asserted by integration tests.

## Internal / Architecture

- The dual-driver `_driver.py` seam introduced in Phase A was removed in Phase G; the codebase is single-driver again.
- `psycopg2-binary` and `types-psycopg2` removed from `pyproject.toml`.
- Type annotations across `src/moncpipelib/` and `tests/integration/` updated from `psycopg2.extensions.connection` / `cursor` to `psycopg.Connection` / `psycopg.Cursor` (now generic over row factory).
- CI no longer runs the dual-driver matrix; the integration job runs once against psycopg3.

## Suggested Follow-up (not in 0.32.0)

Two items emerged from the burn-test work that are out of scope for this release but worth filing:

1. **SCD2 index requirements documented.** Add the partial-index requirement to `docs/scd2-guide.md` so consumers don't accidentally ship a 100x-slower merge.
2. **Connection reuse for hot pipelines.** A reuse-conn experiment showed that holding one connection across multiple `database.write()` calls eliminates the per-connection backend warmup tail entirely (psycopg3 reuse-conn median was 3.10s with zero >=10s outliers vs 3.25s median / 3 outliers with fresh-conn). The migration plan deferred `psycopg-pool` integration; the data suggests revisiting it as a follow-on initiative if raw-COPY tail latency becomes a constraint.
